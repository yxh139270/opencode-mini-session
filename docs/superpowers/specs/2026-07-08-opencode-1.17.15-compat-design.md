# OpenCode 1.17.15 Compatibility Design

## 背景

当前插件仓库依赖的是：

- `@opencode-ai/plugin@1.14.48`
- `@opencode-ai/sdk@1.14.48`
- `@opentui/core@0.2.6`
- `@opentui/solid@0.2.6`
- `@opentui/keymap@0.2.6`

而当前运行环境中的 OpenCode 主程序版本已经是 `1.17.15`，npm 上对应的插件相关包版本也为：

- `@opencode-ai/plugin@1.17.15`
- `@opencode-ai/sdk@1.17.15`

同时，`@opencode-ai/plugin@1.17.15` 对 OpenTUI 的 peer 依赖要求已经提升到：

- `@opentui/core >= 0.4.3`
- `@opentui/solid >= 0.4.3`
- `@opentui/keymap >= 0.4.3`

这说明仅仅维持旧依赖并依靠运行时侥幸兼容，并不是稳定方案。仓库里虽然已经存在一定的新旧版本兼容代码，但这部分逻辑目前散落在业务文件中，例如：

- `src/index.ts` 中的 `supportsKeymap` / `api.command` 分支
- `src/version.ts` 中的最低版本判断
- `src/keybinds.ts` 中对不同注册路径的直接依赖

这些分支虽然能解决一部分问题，但随着 OpenCode 继续演进，会让业务代码承担越来越多“平台版本差异”判断。

## 目标

- 兼容 OpenCode `1.17.15` 对应的插件运行环境。
- 保留当前对较旧 OpenCode 版本的 fallback 支持。
- 将 OpenCode 版本差异收口到一个薄兼容层，减少业务代码中的平台判断。
- 在不改变 mini session 现有核心行为的前提下完成升级。

## 非目标

- 不重写 mini session、agent、overlay、session 生命周期逻辑。
- 不借机重构无关文件或调整交互设计。
- 不为了抽象而抽象，不引入大型适配框架。
- 不承诺兼容所有历史 OpenCode 版本，只保留仓库当前已有的旧版 fallback 目标。

## 约束

- 以最小正确改动为原则。
- 兼容层应当很薄，只做能力探测、版本判断和 API 注册路径收口。
- 业务逻辑仍留在原文件中，不把现有功能行为迁移到兼容层。
- 依赖升级后必须通过现有类型检查和测试。

## 方案选型

### 方案 A：仅升级依赖并在业务代码里继续补分支

优点：

- 初始改动最少。

缺点：

- 版本差异继续散落在业务层。
- 未来再升级时，仍要在多个业务文件中搜索和修补兼容点。
- 容易让 `index.ts` 持续膨胀。

### 方案 B：维持旧依赖不动，只做运行时兜底

优点：

- 表面上改动更小。

缺点：

- 与当前主程序版本长期错位。
- 容易再次遇到类型入口、peer 依赖、运行时能力不一致问题。
- 不是稳定维护方案。

### 方案 C：升级依赖，并新增薄兼容层统一收口版本差异

优点：

- 同时解决当前 `1.17.15` 兼容问题和未来维护成本问题。
- 业务代码只依赖统一后的能力接口，更稳定、更易读。
- 改动规模仍可控，不需要大面积重写。

缺点：

- 比单纯升级依赖多出一层抽象设计。

### 结论

采用方案 C。

## 设计

### 依赖升级

将仓库依赖升级到与当前主程序相匹配的版本：

- `@opencode-ai/plugin` 升级到 `1.17.15`
- `@opencode-ai/sdk` 升级到 `1.17.15`
- `@opentui/core`、`@opentui/solid`、`@opentui/keymap` 升级到满足 `1.17.15` peer 依赖的版本，至少 `0.4.3`

升级后以实际类型错误和测试结果为准做最小修复，不预设额外行为变更。

### 新增兼容层

新增一个单独文件，例如：`src/opencode-compat.ts`。

它的职责是：

- 封装 OpenCode 能力探测
- 封装版本判断
- 封装 keymap 与 legacy command 注册路径
- 为业务层暴露更稳定的统一接口

它不负责：

- mini session 业务逻辑
- overlay 渲染
- prompt / session / permission 行为
- agent、model、routing 规则

兼容层应当输出少量、明确的方法，例如：

- `supportsKeymap(api): boolean`
- `registerMiniSessionBindings(...)`
- 必要时提供统一的 capability 结果对象

重点是收口差异，而不是复制现有业务逻辑。

### `src/index.ts` 调整

`src/index.ts` 目前同时承担：

- 插件入口
- overlay 信号管理
- 会话关闭副作用
- keymap / legacy command 注册分支

本次只调整其中“版本差异相关”的部分：

- 将 `supportsKeymap`、`api.command` fallback 等判断迁入 compat 层
- `index.ts` 只负责把上下文、action、overlay 状态传给 compat 层
- 现有 `triggerMiniMode()`、`openModelPicker()`、overlay 状态管理保持原样

这样能减少入口文件对不同平台版本细节的直接依赖。

### `src/keybinds.ts` 调整

`src/keybinds.ts` 保留现有动作构造逻辑，但不再承担“是否该走哪条注册路径”的决策。

也就是说：

- 它继续负责生成 actions / commands
- 兼容层负责决定用 `api.keymap` 还是 `api.command` 方式注册

如果某些现有导出函数语义已经明显绑定特定版本实现，可以考虑在不改变行为的前提下做轻微整理，但不做功能性重写。

### 版本知识的收口

仓库中现有的版本知识，如 `MIN_KEYMAP_VERSION`，应尽量移动到 compat 层或由 compat 层统一引用。

原则是：

- 业务文件不再直接关心“从哪个版本开始支持某能力”
- 业务文件只关心“这个能力现在能不能用”

### 行为保持不变

以下行为在本次实现中应保持不变：

- mini session 的打开、关闭、隐藏、恢复
- 主/新鲜模式切换
- 模型选择逻辑
- permission prompt 的隐藏/恢复交互
- session 流式输出与 counter 行为

兼容层只服务于“如何接入宿主 API”，不改变“插件做什么”。

## 错误处理

- 如果升级依赖后出现类型不兼容，优先通过兼容层和最小局部适配解决。
- 如果某个新版本 API 需要更严格的判空或能力探测，应在 compat 层内处理，而不是把保护逻辑扩散到业务层。
- 如果发现旧版 fallback 与新版 API 完全冲突，先保持当前仓库已支持的旧路径不变，再对新版单独补适配，不直接删除旧逻辑。

## 测试与验证

至少覆盖以下验证：

1. `npm install` 后依赖图可正常解析。
2. `npm run typecheck` 通过。
3. `npm test` 通过，特别是：
   - `tests/session.test.ts`
   - `tests/agent.test.ts`
   - `tests/context.test.ts`
4. 兼容层引入后，现有 keymap / legacy 注册逻辑行为没有回归。

如果现有测试不足以覆盖兼容层的注册决策，应补充针对 compat 层或入口层的最小回归测试。

## 风险

- OpenTUI 从 `0.2.x` 升级到 `0.4.x` 可能带来局部类型变化，尤其是渲染组件和事件类型。
- `@opencode-ai/plugin@1.17.15` 可能收紧某些 API 类型，导致当前宽松写法需要补显式保护。
- 如果兼容层边界划分过大，容易演变成不必要重构；如果边界划分过小，又无法真正降低未来维护成本。

## 成功标准

- 仓库依赖成功升级到与 OpenCode `1.17.15` 匹配的版本。
- 插件在新版 OpenCode 中可正常加载。
- 旧版 fallback 逻辑仍被保留。
- 版本差异判断从业务代码中明显收敛到 compat 层。
- 测试与类型检查通过。
